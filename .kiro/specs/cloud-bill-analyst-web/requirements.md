# Requirements Document

## Introduction

Cloud Bill Analyst (Web) is a Next.js App Router web application that serves as the
conversational front end to an already-deployed AWS Bedrock AgentCore runtime (the
Python agent in `agent/`). A signed-in user connects one or more AWS accounts
read-only, then chats with the agent to analyze spend, watch a live "what the agent
is doing" activity timeline, surface cost anomalies, and export polished PDF/Excel
reports.

The agent is only ever invoked server-side. AWS access keys are never collected.
Per-account secrets (`role_arn`, `external_id`) are resolved server-side and never
exposed to the browser. This document specifies the MVP behavior in EARS format and
follows the workspace steering in `.kiro/steering/` and the invocation contract in
`agent/AGENTCORE_INTEGRATION.md`.

Confirmed design decisions (from clarification):
- The dashboard overview and anomaly detection are computed by the app querying AWS
  Cost Explorer directly server-side via the connected account's assumed read-only role.
- Human-in-the-loop approve/reject is a client-side gate the app injects before
  specific actions (first scan of a newly connected account, explicit export requests).
- A new server env var `CBA_RUNTIME_ROLE_ARN` holds the runtime execution role ARN
  used as the trust principal in the generated CloudFormation template.
- Account connection offers both a copyable/downloadable CloudFormation template and a
  "Launch Stack" console deep link.
- `display_currency` and `timezone` are per-account settings defaulting to the contract
  values (`IDR`, `Asia/Jakarta`).
- Each chat thread is pinned to exactly one connected account chosen at creation.
- Message feedback is persisted to Postgres; regenerate re-invokes the agent.
- Registration is open self-serve email/password with no email verification for MVP.

## Glossary

- **Web_App**: The Next.js App Router application defined by this specification.
- **Auth_Service**: The Auth.js-based authentication subsystem (credentials provider,
  Drizzle adapter, database-backed sessions).
- **User**: A person with a registered Web_App account, identified by an application
  user id (`actor_id`).
- **Account_Wizard**: The server-side subsystem that guides a User through connecting
  a read-only cross-account AWS role.
- **Connected_Account**: A stored record of `{ alias, role_arn, encrypted external_id,
  display_currency, timezone }` belonging to one User.
- **Connection_Tester**: The server-side subsystem that assumes a candidate role and
  runs a minimal Cost Explorer query to validate a connection.
- **Chat_Relay**: The Node-runtime server route that invokes `InvokeAgentRuntime` and
  relays the Server-Sent Events (SSE) stream to the browser.
- **Agent_Runtime**: The deployed Bedrock AgentCore runtime identified by
  `CBA_RUNTIME_ARN`.
- **Thread**: A saved chat conversation, pinned to one Connected_Account, mapped to a
  stable `runtimeSessionId`.
- **Session_Id**: The `runtimeSessionId` string (33-128 characters) stable per Thread.
- **Activity_Timeline**: The UI element that renders `tool` SSE events as a
  collapsible step list.
- **Report_Service**: The server-side subsystem that mints short-lived presigned S3
  GET URLs for report objects.
- **Anomaly_Detector**: The server-side subsystem that computes cost anomalies (spikes,
  new services, large month-over-month deltas) from Cost Explorer data.
- **Dashboard**: The `/dashboard` page showing spend overview, connected accounts, and
  anomaly flags.
- **Confirmation_Gate**: The client-side approve/reject prompt injected before
  sensitive actions.
- **Suggestion_Engine**: The subsystem producing recommendation-style, variative prompt
  chips.
- **Cost_Explorer**: The AWS Cost Explorer service (`ce:GetCostAndUsage`,
  `ce:GetDimensionValues`, `ce:GetCostForecast`).
- **External_Id**: A per-account secret generated by the Web_App, stored encrypted with
  `APP_ENCRYPTION_KEY`, used in the role trust condition and invocation context.
- **SSE_Event**: A streamed event of type `delta`, `tool`, `report_file`, `error`, or
  `done` from the Agent_Runtime.

## Requirements

### Requirement 1: User Registration

**User Story:** As a visitor, I want to register with email and password, so that I can access the application.

#### Acceptance Criteria

1. WHEN a visitor submits the registration form with a syntactically valid email of at most 254 characters and a password between 8 and 128 characters inclusive, THE Auth_Service SHALL create exactly one User record with the password stored as an argon2 hash and return a success indication within 5 seconds.
2. WHEN a visitor submits the registration form with an email whose normalized (case-insensitive, trimmed) value matches an existing User, THE Auth_Service SHALL reject the registration, create no new User record, and return a message stating the email is already in use.
3. IF a visitor submits the registration form with an email that is empty, exceeds 254 characters, or does not match a standard email address format (local-part@domain), THEN THE Auth_Service SHALL reject the submission, persist no data, and return a validation message identifying the email field as invalid.
4. IF a visitor submits the registration form with a password shorter than 8 characters or longer than 128 characters, THEN THE Auth_Service SHALL reject the submission, persist no data, and return a validation message identifying the password field as invalid.
5. THE Auth_Service SHALL store the plaintext password in no persistent store.
6. WHEN a User record is created successfully, THE Auth_Service SHALL establish an authenticated database-backed session bound to that User's id and return the session identifier to the client.

### Requirement 2: User Login and Sessions

**User Story:** As a registered user, I want to log in and stay signed in, so that I can use the application across page loads.

#### Acceptance Criteria

1. WHEN a User submits the login form with an email and password that match a stored User record, THE Auth_Service SHALL establish an authenticated session persisted in PostgreSQL within 3 seconds and redirect the request to the dashboard page.
2. IF a User submits the login form with an email or password that does not match a stored User record, THEN THE Auth_Service SHALL reject the attempt without creating a session, retain the entered email value in the form, and return a generic message stating the credentials are invalid without indicating whether the email or the password was incorrect.
3. IF a User submits the login form with an empty email field or an empty password field, THEN THE Auth_Service SHALL reject the attempt without creating a session and return a message indicating the missing field.
4. WHEN an unauthenticated request targets a guarded route, THE Web_App SHALL redirect the request to the `/login` page.
5. WHEN an authenticated User requests to sign out, THE Auth_Service SHALL delete the User session record from PostgreSQL and redirect the request to the `/login` page.
6. THE Auth_Service SHALL associate each authenticated session with the User application id used as `actor_id` in Agent_Runtime invocations.
7. WHILE an authenticated session exists in PostgreSQL and has not exceeded its 30-day maximum lifetime, THE Web_App SHALL treat requests bearing that session as authenticated across page loads without re-prompting for credentials.
8. WHEN an authenticated session reaches its 30-day maximum lifetime, THE Auth_Service SHALL invalidate the session, delete the session record from PostgreSQL, and require the User to re-authenticate on the next request to a guarded route.
9. IF a User submits 5 consecutive failed login attempts for the same email within a 15-minute window, THEN THE Auth_Service SHALL reject further login attempts for that email for 15 minutes and return a message indicating too many failed attempts.

### Requirement 3: Connect an AWS Account via Wizard

**User Story:** As a signed-in user, I want a guided wizard to connect my AWS account read-only, so that the agent can analyze my spend without me sharing AWS keys.

#### Acceptance Criteria

1. WHEN a User starts the Account_Wizard, THE Account_Wizard SHALL generate an External_Id for the pending connection that is between 16 and 1224 characters in length and unique across all existing and pending connections.
2. THE Account_Wizard SHALL present a CloudFormation template that provisions an AWS IAM role granting exactly the three read-only actions `ce:GetCostAndUsage`, `ce:GetDimensionValues`, and `ce:GetCostForecast` and no other actions, with a trust policy whose principal is the value of `CBA_RUNTIME_ROLE_ARN` and whose condition requires `sts:ExternalId` to equal the generated External_Id.
3. THE Account_Wizard SHALL provide the CloudFormation template as both copyable inline text and a downloadable file, and SHALL provide a "Launch Stack" AWS console deep link that references the template.
4. THE Account_Wizard SHALL collect an AWS account role ARN pasted by the User and SHALL collect an account alias that is between 1 and 100 characters in length after leading and trailing whitespace is trimmed.
5. THE Account_Wizard SHALL collect no AWS access key id and no AWS secret access key at any step.
6. IF the pasted role ARN fails IAM role ARN format validation, THEN THE Account_Wizard SHALL reject the input, retain all values entered by the User, and return an error indication identifying the invalid ARN.
7. IF the role ARN or account alias is empty or contains only whitespace, THEN THE Account_Wizard SHALL reject the submission, retain all values entered by the User, and return a field-specific error indication identifying which field is empty.
8. WHEN the User submits a role ARN that passes IAM role ARN format validation and an account alias between 1 and 100 characters after trimming, THE Account_Wizard SHALL associate the role ARN, account alias, and generated External_Id with the pending connection.

### Requirement 4: Test and Store a Connected Account

**User Story:** As a signed-in user, I want the app to test my connection before saving it, so that I know the role works before I rely on it.

#### Acceptance Criteria

1. WHEN a User requests "Test connection" for a pending connection, THE Connection_Tester SHALL assume the pasted role using the generated External_Id and SHALL execute a minimal Cost_Explorer `GetCostAndUsage` query scoped to a single day with granularity DAILY and a single metric.
2. IF the pasted role ARN does not match the AWS IAM role ARN format, or the alias is empty or exceeds 100 characters, THEN THE Connection_Tester SHALL reject the request with a validation error indicating the offending field, and THE Account_Wizard SHALL NOT attempt an assume-role.
3. IF the Connection_Tester assume-role and query do not both complete successfully within 30 seconds, THEN THE Connection_Tester SHALL treat the test as failed.
4. WHEN the Connection_Tester assume-role and query both succeed within 30 seconds, THE Account_Wizard SHALL store exactly one Connected_Account containing the alias, the role ARN, the External_Id encrypted with `APP_ENCRYPTION_KEY`, a `display_currency` equal to `IDR`, and a `timezone` equal to `Asia/Jakarta`.
5. IF the Connection_Tester assume-role fails, the query fails, or the 30-second limit is exceeded, THEN THE Connection_Tester SHALL return a redacted error message that describes the failure category without exposing the External_Id, role credentials, or internal identifiers, and THE Account_Wizard SHALL NOT store a Connected_Account and SHALL leave any prior stored accounts unchanged.
6. THE Account_Wizard SHALL store the External_Id only in encrypted form and SHALL include the decrypted External_Id in no HTTP response sent to the browser.
7. WHEN a Connected_Account is stored, THE Web_App SHALL associate that Connected_Account with the currently authenticated User.

### Requirement 5: Manage Multiple Connected Accounts

**User Story:** As a signed-in user, I want to connect several AWS accounts and switch between them, so that I can analyze each account's spend.

#### Acceptance Criteria

1. THE Web_App SHALL allow a User to store between 1 and 10 Connected_Accounts, inclusive.
2. IF a User requests to store a Connected_Account when that User already has 10 Connected_Accounts, THEN THE Web_App SHALL reject the request, retain the existing Connected_Accounts unchanged, and return an error indicating the maximum account limit has been reached.
3. WHEN a User views the `/accounts` page, THE Web_App SHALL list each of that User's Connected_Accounts showing the account alias (1 to 100 characters) and a masked account identifier that reveals only the last 4 digits of the 12-digit AWS account number with all preceding digits replaced by a masking character.
4. WHEN a User with zero Connected_Accounts views the `/accounts` page, THE Web_App SHALL display an empty-state message indicating no accounts are connected.
5. WHEN a User selects a Connected_Account in the account switcher, THE Web_App SHALL set that Connected_Account as the active account for all subsequent account-scoped actions and SHALL persist the active-account selection across the User's sessions until the User selects a different Connected_Account.
6. WHEN a User confirms removal of a Connected_Account, THE Web_App SHALL delete that Connected_Account record, including its encrypted External_Id, from persistent storage within 5 seconds and return confirmation that the deletion succeeded.
7. IF a User removes the Connected_Account that is currently the active account, THEN THE Web_App SHALL clear the active-account selection, and WHERE at least one other Connected_Account remains, THE Web_App SHALL require the User to select a new active account before performing further account-scoped actions.
8. IF deletion of a Connected_Account fails, THEN THE Web_App SHALL retain the Connected_Account record unchanged and return an error indicating the removal did not complete.
9. THE Web_App SHALL exclude any decrypted External_Id from every account listing response, account switcher response, and account detail response returned to the browser.

### Requirement 6: Chat Availability Gating

**User Story:** As a signed-in user, I want the chat to be usable only after I connect an account, so that I am not confused by a chat that cannot work.

#### Acceptance Criteria

1. WHILE the authenticated User has zero Connected_Accounts, THE Web_App SHALL render the chat composer in a disabled state that prevents text entry and prevents submission, and SHALL display a keyboard-reachable call-to-action to connect an account.
2. WHILE the authenticated User has at least one Connected_Account, THE Web_App SHALL render the chat composer in an enabled state that accepts text entry and submission.
3. WHEN the authenticated User connects their first Connected_Account, THE Web_App SHALL transition the chat composer from the disabled state to the enabled state within 2 seconds without requiring a full page reload.
4. WHEN the authenticated User disconnects their last Connected_Account, THE Web_App SHALL transition the chat composer from the enabled state to the disabled state within 2 seconds and SHALL display the call-to-action to connect an account.
5. IF a chat submission request is received for a User with zero Connected_Accounts, THEN THE Chat_Relay SHALL reject the request without forwarding it to the agent runtime, SHALL return an error indicating that an account must be connected, and SHALL preserve the User's composed message text in the composer.

### Requirement 7: Server-Side Agent Invocation and SSE Relay

**User Story:** As a signed-in user, I want my chat messages answered by the agent with streamed output, so that I get timely, incremental responses.

#### Acceptance Criteria

1. WHEN an authenticated User submits a chat prompt for a Thread, THE Chat_Relay SHALL invoke `InvokeAgentRuntime` with accept type `text/event-stream`, using the runtime ARN read from `process.env.CBA_RUNTIME_ARN`.
2. THE Chat_Relay SHALL run on the Node runtime and SHALL respond with `Content-Type: text/event-stream` and `Cache-Control: no-cache`, with response buffering disabled such that each received SSE_Event is flushed to the browser within 1 second of receipt from the Agent_Runtime.
3. THE Chat_Relay SHALL set the invocation `context.actor_id` to the authenticated User application id.
4. THE Chat_Relay SHALL resolve `role_arn` and the decrypted External_Id for the Thread's pinned Connected_Account server-side, and SHALL ensure that no SSE relay output sent to the browser (including `delta`, `tool`, `report_file`, `error`, and `done` events) contains `role_arn` or `external_id`.
5. THE Chat_Relay SHALL set `context.display_currency` and `context.timezone` from the Thread's pinned Connected_Account settings, defaulting to `IDR` and `Asia/Jakarta` respectively when those settings are unset.
6. WHEN the Chat_Relay receives SSE_Events from the Agent_Runtime, THE Chat_Relay SHALL forward `delta`, `tool`, `report_file`, `error`, and `done` events to the browser in received order, and SHALL keep the stream open until a `done` or `error` event is forwarded or until 120 seconds elapse with no new event received.
7. IF the Chat_Relay receives an SSE_Event whose type is none of `delta`, `tool`, `report_file`, `error`, or `done`, THEN THE Web_App SHALL ignore that event and SHALL continue processing subsequent events.
8. IF the Agent_Runtime invocation fails to start, THEN THE Chat_Relay SHALL send exactly one `error` event to the browser whose message describes the failure without containing `role_arn`, `external_id`, AWS credentials, or the runtime ARN, and SHALL then close the stream.
9. THE Chat_Relay SHALL set `runtimeSessionId` to a value of 33 to 128 characters that is identical across all invocations for the same Thread and distinct for every different Thread.
10. IF a chat prompt is submitted by a request without a valid authenticated session, THEN THE Chat_Relay SHALL reject the request without invoking `InvokeAgentRuntime` and without emitting any SSE relay output.
11. IF a chat prompt is submitted for a Thread that has no pinned Connected_Account, THEN THE Chat_Relay SHALL reject the request without invoking `InvokeAgentRuntime` and SHALL return an error response indicating that a connected account is required.

### Requirement 8: Thread and Session Continuity

**User Story:** As a signed-in user, I want my conversations saved and remembered by the agent, so that follow-up questions retain context.

#### Acceptance Criteria

1. WHEN a User creates a Thread with exactly one Connected_Account selected, THE Thread_Manager SHALL pin the Thread to that Connected_Account for the Thread lifetime.
2. IF a User attempts to create a Thread with zero Connected_Accounts selected, THEN THE Thread_Manager SHALL reject the creation, SHALL NOT persist any Thread or Session_Id, and SHALL return an error indicating that at least one Connected_Account is required.
3. WHEN a Thread is created, THE Thread_Manager SHALL generate a Session_Id of length between 33 and 128 characters inclusive and SHALL persist a one-to-one Thread-to-Session_Id mapping.
4. WHILE a Thread exists, THE Thread_Manager SHALL supply the same persisted Session_Id for every invocation of that Thread and SHALL NOT reassign or alter that Session_Id for the Thread lifetime.
5. WHEN a User who owns a Thread opens that previously saved Thread, THE Web_App SHALL display that Thread's persisted messages ordered by message creation timestamp ascending, oldest message first.
6. WHEN a User who owns a Thread opens that Thread and the Thread contains zero messages, THE Web_App SHALL display the empty Thread without surfacing an error.
7. IF a User requests a Thread that the User does not own, THEN THE Web_App SHALL deny access and SHALL NOT expose that Thread's messages or Session_Id.
8. THE Web_App SHALL allow an authenticated User to have one or more saved Threads.
9. WHEN a User views the Thread list, THE Web_App SHALL list only Threads belonging to the authenticated User.

### Requirement 9: Live Activity Timeline

**User Story:** As a signed-in user, I want to see what the agent is doing while it works, so that I trust and understand the response.

#### Acceptance Criteria

1. WHEN the Web_App receives a `tool` event with `phase` equal to `start` and an `id` not matching any existing step, THE Activity_Timeline SHALL append a new step, appended below any previously added steps in received order, displaying the event `status` text, the event `label` as a badge, and an animated spinner indicator, within 200 milliseconds of event receipt.
2. IF the Web_App receives a `tool` event with `phase` equal to `start` whose `id` matches an existing step, THEN THE Activity_Timeline SHALL update that existing step's `status` text and `label` badge in place without appending a duplicate step.
3. WHEN the Web_App receives a `tool` event with `phase` equal to `end` whose `id` matches an existing step, THE Activity_Timeline SHALL replace that step's spinner indicator with a check-mark indicator and retain the step's `status` text and `label` badge.
4. IF the Web_App receives a `tool` event with `phase` equal to `end` whose `id` matches no existing step, THEN THE Activity_Timeline SHALL ignore the event, leave all existing steps unchanged, and continue processing subsequent events.
5. WHILE a step has received a `start` event and no `end` event with a matching `id`, THE Activity_Timeline SHALL display that step's animated spinner indicator and no check-mark indicator.
6. WHEN the Web_App receives a `done` event, THE Activity_Timeline SHALL collapse all steps into a single-line summary that concatenates each completed step's `status` (or `label`) in received order, and SHALL provide a User-activatable control that re-expands the full step list.
7. IF the Web_App receives an `error` event while one or more steps have a `start` event and no matching `end` event, THEN THE Activity_Timeline SHALL stop each such step's spinner indicator and mark it as not completed, retaining its `status` text and `label` badge.
8. WHEN a step is appended or its indicator changes, THE Activity_Timeline SHALL update an `aria-live="polite"` region with the affected step's current `status` text.

### Requirement 10: Streamed Answer Rendering

**User Story:** As a signed-in user, I want the agent's answer rendered as readable markdown, so that cost tables and code are legible.

#### Acceptance Criteria

1. WHEN the Web_App receives a `delta` event, THE Web_App SHALL append the event `text` to the in-progress assistant message in the order the events are received over the SSE stream, updating the rendered content within 200 ms of receipt.
2. WHEN the Web_App displays assistant message content, THE Web_App SHALL render it as markdown supporting at minimum GitHub-flavored markdown tables and inline code chips.
3. WHILE assistant `delta` events are streaming AND the user has not manually scrolled away from the latest content, THE Web_App SHALL keep the message list scroll position anchored to the most recently appended content while preserving all previously rendered content.
4. WHEN the Web_App renders a user turn, THE Web_App SHALL display it as a right-aligned message element.
5. WHEN the Web_App renders an assistant turn, THE Web_App SHALL display it as a left-aligned message element.
6. IF the user manually scrolls away from the latest content WHILE assistant `delta` events are streaming, THEN THE Web_App SHALL stop auto-scrolling and retain the user's current scroll position until the user scrolls back to the latest content.
7. IF a received `delta` event contains malformed or incomplete markdown, THEN THE Web_App SHALL render the accumulated message text without discarding earlier content and without surfacing an unhandled error to the user.

### Requirement 11: Report Download

**User Story:** As a signed-in user, I want to download reports the agent produces, so that I can share polished PDF and Excel outputs.

#### Acceptance Criteria

1. WHEN the Web_App receives a `report_file` event, THE Report_Service SHALL mint a presigned S3 GET URL for the event `key` using `CBA_REPORT_BUCKET` server-side.
2. THE Report_Service SHALL set the presigned URL expiry to a value between 1 and 300 seconds inclusive.
3. THE Report_Service SHALL authorize a report `key` only when it has the exact form `cloud-bill-analyst/reports/<actor_id>/<filename>`, where `<actor_id>` equals the authenticated User's `actor_id` as the path segment immediately following the fixed `cloud-bill-analyst/reports/` prefix (not merely a leading or contained substring). IF the `key` does not match this form for the authenticated User, THEN THE Report_Service SHALL reject the request, return an authorization error, and SHALL NOT mint or return any presigned URL.
4. IF the report `key` or `CBA_REPORT_BUCKET` is missing, or the presign operation fails, THEN THE Report_Service SHALL return an error indicating the presign failure and SHALL NOT return a partial or malformed presigned URL.
5. WHILE the presigned URL for a received `report_file` event is not yet available, THE Web_App SHALL NOT render the report download card.
6. WHEN the presigned URL becomes available, THE Web_App SHALL render the report download card with a download control and a file-type indicator that reflects whether the report `key` denotes a PDF or an XLSX file.

### Requirement 12: Dashboard Overview

**User Story:** As a signed-in user, I want a dashboard summarizing my spend and connected accounts, so that I get an at-a-glance overview.

#### Acceptance Criteria

1. WHEN an authenticated User with at least one Connected_Account opens `/dashboard`, THE Dashboard SHALL query Cost_Explorer server-side using the active Connected_Account's assumed role, where the active Connected_Account is the most recently selected Connected_Account, defaulting to the first Connected_Account when none has been selected.
2. WHEN the Cost_Explorer query completes successfully within 10 seconds, THE Dashboard SHALL display a spend overview for the active Connected_Account, where the spend overview is the total spend for the current calendar month to date expressed in the account's configured display currency.
3. WHILE the Cost_Explorer query is in progress, THE Dashboard SHALL display a loading state in place of the spend overview.
4. THE Dashboard SHALL display the authenticated User's Connected_Accounts.
5. WHILE the authenticated User has zero Connected_Accounts, THE Dashboard SHALL display a call-to-action to connect an account in place of the spend overview and SHALL NOT query Cost_Explorer.
6. IF the Cost_Explorer query for the Dashboard fails or does not complete within 10 seconds, THEN THE Dashboard SHALL display a redacted error state in place of the spend overview, indicating the query failed, and SHALL provide a retry control to re-run the query.

### Requirement 13: Cost Anomaly Detection and Flags

**User Story:** As a signed-in user, I want unusual spend flagged, so that I can spot spikes, new services, and large month-over-month increases.

#### Acceptance Criteria

1. WHEN the Anomaly_Detector evaluates a Connected_Account, THE Anomaly_Detector SHALL query Cost_Explorer server-side using that account's assumed role, complete the evaluation within 10 seconds, and classify each detected anomaly as exactly one of: spike, new service, or large month-over-month delta.
2. WHERE the Anomaly_Detector detects at least one anomaly for the active Connected_Account, THE Dashboard SHALL display one anomaly badge per detected anomaly, and each badge SHALL indicate the anomaly's classification (spike, new service, or large month-over-month delta).
3. WHERE the Anomaly_Detector detects at least one anomaly relevant to the current conversation, THE Web_App SHALL render one inline anomaly callout per anomaly in the chat, using a rose accent for a spike and an amber accent for a new service or a large month-over-month delta.
4. THE Anomaly_Detector SHALL classify a month-over-month cost increase as a large delta only when the increase, computed against the immediately preceding full calendar month, meets or exceeds 25 percent.
5. THE Anomaly_Detector SHALL classify a service as a spike only when its cost for a single day meets or exceeds 50 percent above that service's trailing 7-day average daily cost.
6. THE Anomaly_Detector SHALL classify a service as a new service only when it has a cost greater than zero in the current calendar month and a cost of zero in the immediately preceding full calendar month.
7. IF the Cost_Explorer query for anomaly detection fails, THEN THE Anomaly_Detector SHALL return zero anomalies, THE Dashboard SHALL display no anomaly badge for that Connected_Account, and THE Web_App SHALL render no inline anomaly callout for that Connected_Account.

### Requirement 14: Message Actions

**User Story:** As a signed-in user, I want to copy, regenerate, and rate assistant messages, so that I can reuse and improve responses.

#### Acceptance Criteria

1. WHEN a User activates the copy action on an assistant message, THE Web_App SHALL write that message's complete rendered text content to the system clipboard and display a copy-confirmation indication that remains visible for at least 2 seconds.
2. IF the copy action is activated but the clipboard write does not succeed, THEN THE Web_App SHALL display an error indication that the copy failed and SHALL leave the message content unchanged.
3. WHEN a User activates the regenerate action on an assistant message for which a preceding User prompt exists in the Thread, THE Chat_Relay SHALL re-invoke the Agent_Runtime for that Thread using the preceding User prompt and the Thread's existing Session_Id.
4. IF a User activates the regenerate action on an assistant message that has no preceding User prompt in the Thread, THEN THE Web_App SHALL disable the regenerate action for that message and SHALL NOT invoke the Agent_Runtime.
5. WHEN a User submits thumbs-up or thumbs-down feedback on an assistant message, THE Web_App SHALL persist exactly one feedback value (either thumbs-up or thumbs-down) for that message in PostgreSQL, replacing any previously stored feedback value for the same message.
6. WHEN a User activates the feedback control that matches the message's currently stored feedback value, THE Web_App SHALL remove the stored feedback value for that message in PostgreSQL and return the message to the no-feedback state.
7. IF persisting or removing a feedback value does not succeed, THEN THE Web_App SHALL display an error indication and SHALL retain the previously stored feedback state.
8. WHEN a User views an assistant message that has a stored feedback value, THE Web_App SHALL display the corresponding thumbs-up or thumbs-down state as selected; otherwise THE Web_App SHALL display both feedback controls in an unselected default state.

### Requirement 15: Human-in-the-Loop Confirmation Gate

**User Story:** As a signed-in user, I want to confirm sensitive actions before they run, so that I stay in control of scans and exports.

#### Acceptance Criteria

1. WHEN a User initiates the first scan of a newly connected account, THE Confirmation_Gate SHALL present an inline approve/reject prompt within 500 milliseconds and SHALL block the Chat_Relay from invoking the Agent_Runtime until the User selects approve or reject.
2. WHEN a User submits an explicit report export request, THE Confirmation_Gate SHALL present an inline approve/reject prompt within 500 milliseconds and SHALL block the Chat_Relay from invoking the Agent_Runtime until the User selects approve or reject.
3. WHEN a User approves a Confirmation_Gate prompt, THE Chat_Relay SHALL invoke the Agent_Runtime for the pending action exactly once, and THE Confirmation_Gate SHALL dismiss the prompt.
4. WHEN a User rejects a Confirmation_Gate prompt, THE Web_App SHALL cancel the pending action, SHALL dismiss the prompt, and SHALL NOT invoke the Agent_Runtime for the cancelled action.
5. WHILE a Confirmation_Gate prompt remains unanswered, THE Confirmation_Gate SHALL keep the pending action blocked and SHALL NOT auto-approve the action or invoke the Agent_Runtime.

### Requirement 16: Suggestion Chips

**User Story:** As a signed-in user, I want recommendation-style prompt suggestions, so that I can quickly ask useful questions.

#### Acceptance Criteria

1. WHEN the Web_App renders a chat empty state or an idle composer, THE Suggestion_Engine SHALL present between 3 and 6 recommendation-style prompt chips, each containing between 1 and 120 characters of prompt text.
2. WHEN a User activates a suggestion chip, THE Web_App SHALL replace the entire current composer contents with that chip's prompt text, move keyboard focus to the composer, and leave the text editable without automatically submitting it.
3. WHEN the Suggestion_Engine presents prompt chips on a render, THE Suggestion_Engine SHALL ensure that at least half of the presented chips differ in wording from the chips presented on the immediately preceding render within the same thread.
4. IF the Suggestion_Engine cannot produce at least 3 valid prompt chips for a render, THEN THE Web_App SHALL render the chat empty state or idle composer without any suggestion chips and SHALL leave the composer usable for free-form input.

### Requirement 17: Per-Account Currency and Timezone Settings

**User Story:** As a signed-in user, I want to set the display currency and timezone per account, so that reports and analysis match my preferences.

#### Acceptance Criteria

1. WHEN a User views a Connected_Account's settings, THE Web_App SHALL display that Connected_Account's `display_currency` and `timezone`, using `IDR` for `display_currency` and `Asia/Jakarta` for `timezone` when either value is unset, within 2 seconds of the settings view being requested.
2. WHEN a User submits an update to a Connected_Account's `display_currency` with a valid ISO 4217 three-letter currency code, or to its `timezone` with a valid IANA time zone identifier, THE Web_App SHALL persist the updated value for that Connected_Account only, without altering the settings of any other Connected_Account, and confirm the update to the User.
3. IF a User submits a `display_currency` value that is not a valid ISO 4217 three-letter currency code, or a `timezone` value that is not a valid IANA time zone identifier, THEN THE Web_App SHALL reject the update, retain the previously persisted value, and present an error indication stating the value is invalid.
4. WHEN the Chat_Relay invokes the Agent_Runtime for a Thread, THE Chat_Relay SHALL set `context.display_currency` and `context.timezone` from the Thread's pinned Connected_Account settings.
5. IF the Thread's pinned Connected_Account has an unset `display_currency` or `timezone` when the Chat_Relay invokes the Agent_Runtime, THEN THE Chat_Relay SHALL set `context.display_currency` to `IDR` and/or `context.timezone` to `Asia/Jakarta` for each unset value respectively.

### Requirement 18: Secret Handling and Server-Side Boundaries

**User Story:** As a security-conscious user, I want secrets kept on the server, so that my AWS access is never exposed through the browser.

#### Acceptance Criteria

1. THE Web_App SHALL execute every AWS SDK call (including `InvokeAgentRuntime`, S3 presign, and STS AssumeRole operations) exclusively within server-side modules that are never imported into client components.
2. WHEN the Web_App composes any HTTP response, SSE event payload, or serialized data delivered to the browser, THE Web_App SHALL exclude the `role_arn` field, the decrypted External_Id value, and any AWS access key, secret key, or session token.
3. THE Web_App SHALL resolve the Agent_Runtime ARN solely by reading `process.env.CBA_RUNTIME_ARN` at request time, and THE Web_App source SHALL contain zero string literals matching the runtime ARN format (`arn:aws:bedrock-agentcore:*`).
4. IF `process.env.CBA_RUNTIME_ARN` is unset or empty when an AgentCore invocation is attempted, THEN THE Web_App SHALL abort the invocation, return a typed server-side error indicating missing runtime configuration, and make no AWS SDK call.
5. WHEN the Web_App persists a connected account, THE Web_App SHALL store the External_Id encrypted at rest using `APP_ENCRYPTION_KEY`, and THE Web_App SHALL retain no plaintext External_Id in the datastore.
6. WHEN a server route receives input, THE Web_App SHALL validate the input against a zod schema before executing any AWS SDK call, database write, or secret access.
7. IF a server route input fails zod schema validation, THEN THE Web_App SHALL reject the request, return a typed validation error identifying the failing field(s) without echoing secret values, and perform no AWS SDK call, database write, or secret access for that request.

### Requirement 19: Environment Configuration

**User Story:** As a developer, I want environment variables documented and safely defaulted, so that I can configure the app without leaking secrets.

#### Acceptance Criteria

1. WHEN the server process starts, THE Web_App SHALL read `DATABASE_URL`, `AUTH_SECRET`, `APP_ENCRYPTION_KEY`, `AWS_REGION`, `CBA_RUNTIME_ARN`, `CBA_RUNTIME_ROLE_ARN`, and `CBA_REPORT_BUCKET` exclusively from the server-side environment and SHALL NOT expose any of these values to the browser.
2. THE Web_App SHALL provide a committed `.env.example` file that contains exactly one placeholder entry for each of the seven required environment variables (`DATABASE_URL`, `AUTH_SECRET`, `APP_ENCRYPTION_KEY`, `AWS_REGION`, `CBA_RUNTIME_ARN`, `CBA_RUNTIME_ROLE_ARN`, `CBA_REPORT_BUCKET`), where each entry holds a non-secret example value.
3. THE Web_App SHALL keep real deployed environment values in a `.env` file that is git-ignored, such that `git status` reports the file as ignored and never staged.
4. WHEN a server route is invoked with all of its dependency environment variables present and non-empty, THE Web_App SHALL process the request without producing any environment-related error.
5. IF a required environment variable is absent or empty when a server route that depends on it is invoked, THEN THE Web_App SHALL return a server error and SHALL record exactly one log entry naming the missing variable, with no environment variable values included in the response or the log entry.

### Requirement 20: Design System and Accessibility Compliance

**User Story:** As a signed-in user, I want a premium, accessible, editorial interface, so that the product feels considered and is usable by everyone.

#### Acceptance Criteria

1. THE Web_App SHALL render headings in Noto Serif, body text in Lora, the Violet accent color, Zinc neutral colors, zero-radius (0px) corners on all cards and containers, HugeIcons line-style icons, and flat surfaces with no gradient backgrounds.
2. THE Web_App SHALL provide a light theme and a dark theme, and WHEN the operating system reports a preferred color scheme with no explicit in-app override set, THE Web_App SHALL apply the theme matching that preference.
3. WHEN the user selects a theme within the Web_App, THE Web_App SHALL apply the selected theme and persist that selection across page reloads within the same authenticated session.
4. WHILE the operating system requests reduced motion (prefers-reduced-motion: reduce), THE Web_App SHALL suppress all non-essential enter animations and scroll-reveal animations, where non-essential means any animation not required to convey a state change (loading, error, or completion).
5. THE Web_App SHALL make the chat composer, message actions, account switcher, and thread list reachable via Tab key traversal, activatable via Enter or Space, and SHALL display a visible focus indicator on the currently focused control.
6. THE Web_App SHALL render text and interactive-control colors at a contrast ratio of at least 4.5:1 for text smaller than 18pt (or 14pt bold) and at least 3:1 for larger text and for control boundaries, in both light and dark themes.
7. WHILE an assistant response is streaming, THE Web_App SHALL announce streaming status updates through an aria-live region set to "polite".
8. WHERE a preset token from the "Sera" preset conflicts with a token from an applied design skill, THE Web_App SHALL apply the "Sera" preset token.
